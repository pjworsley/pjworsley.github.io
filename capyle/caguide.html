Introduction

CAPyLE is composed of two main parts: 1. The cellular automata framework 2. The GUI used to experiment with the CA

The cellular automata (CA) framework compromises of a number of Classes, when paired with a CA description will run a CA and output its grid state for each timestep to a file. This framework's usefulness is reliant on the speed of execution and hence is heavily biased towards performance over readability; numpy arrays are used throughout and unless you have a good understanding of numpy array operations, it may be a little confusing as to what is going on at first. These steps to increase performance were necessary as in a compiled language like C or Java, you may be able to get away with:

for (row in grid){
  for (col in row){
    do x;
  }
}

Whereas in Python, an interpreted language, such operations do not scale well, and we must look for alternatives; in this case numpy arrays.

The GUI component is designed to be used only with the CA implementation above and allows the user to interact with the CA's execution conditions and allows the user to experiment and observe. The GUI framework used is Tkinter (python's inbuilt GUI framework) and although other frameworks are more advanced and look better, the target computer systems did not allow the installation of extra librairies and such Tkinter was used inconjucntion with matplotlip for plotting graphs (CA output).

Module layout

A Python module is a collection of code that a user may import and use the classes and definitions much like using an API. This project has one main module 'capyle' which houses some classes and two submodules which also contain class definitions. Below is the module structure of capyle.

| _PlaybackControls

    | _ScreenshotUI
    | _Separator
    | _StateColorsUI

The underscore leading a class name indicates an internal (The non enforced version of Java's private') class

The important items to the user are Display, the contents of capyle.ca and utils. Display is the class that simply has to be initialised to create the main GUI ready to accept and run a CA description. The non-internal classes of capyle.ca contains the classes used when creating one of these descriptions, for example a Grid1D object requires an instance of Neighbourhood as one of its parameters.
